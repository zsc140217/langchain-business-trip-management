## 🔑 核心创新点

### 1. 混合复杂度评估（解决弱模型工具调用问题）

**问题**：通义千问等弱模型在多工具场景下工具调用成功率为 0%

**解决方案**：
- 80% 使用基于规则的分类（<1ms 延迟）
  - 意图计数（天气、客户、路线、酒店、政策关键词）
  - 实体提取（城市数量）
  - 规划关键词检测
- 20% 使用 LLM 确认（1-2s 延迟，仅用于边缘情况）

**结果**：
- ✅ 工具调用成功率：0% → 100%
- ✅ 准确率：90%
- ✅ 平均延迟：<500ms
- ✅ 成本节省：80%（相比纯 LLM 路由）

### 2. 三路混合检索（80% 准确率）

**架构**：
```
用户查询："魔都出差住宿能报多少"
    ↓
[查询重写] 少样本提示标准化
    原始："魔都出差住宿能报多少"
    重写："上海一类城市出差住宿费用报销标准"
    ↓
[三路并行检索]
    ├─ BM25 稀疏检索（精确关键词匹配，jieba 分词）
    ├─ 稠密检索-原始（语义理解）
    └─ 稠密检索-重写（改进召回率）
    ↓
[RRF 融合] score(doc) = Σ weight_i / (60 + rank_i)
    ↓
Top-K 文档（K=3）
```

**性能对比**：
- BM25 单独：50% 准确率
- 稠密检索单独：60% 准确率
- 三路混合 + RRF：**80% 准确率**
- 查询重写：+20-30% 召回率提升

### 3. 任务分解与并行执行

**流程**：
```python
用户查询："规划杭州出差：查天气、推荐酒店、找客户地址"
    ↓
[TaskDecomposer] LLM 生成结构化 JSON
    [
        {id: 0, type: "QUERY_WEATHER", params: {city: "杭州"}, depends_on: []},
        {id: 1, type: "QUERY_HOTEL", params: {city: "杭州"}, depends_on: []},
        {id: 2, type: "QUERY_CUSTOMER", params: {keyword: "阿里巴巴"}, depends_on: []},
        {id: 3, type: "QUERY_ROUTE", params: {...}, depends_on: [1,2]}
    ]
    ↓
[拓扑排序] 解析依赖关系
    批次 1: [任务 0, 1, 2] (无依赖，并行)
    批次 2: [任务 3] (依赖 1 和 2，顺序)
    ↓
[AsyncIO 并行执行]
    批次 1: asyncio.gather([execute(0), execute(1), execute(2)])
    → 50% 时间节省 vs 顺序执行
    批次 2: execute(3) 使用批次 1 的结果
    ↓
[LLM 整合] 合成连贯响应
```

### 4. 三层记忆系统

| 层级 | 存储方式 | 时间范围 | 用途 |
|------|---------|---------|------|
| **Layer 1: ChatMemory** | 文件持久化 | 20 条消息滑动窗口 | 即时对话上下文 |
| **Layer 2: WorkingMemory** | 内存 | 30 分钟 TTL | 实时实体提取 |
| **Layer 3: LongTermMemory** | JSON 用户画像 | 永久（增量学习） | 个性化推荐 |

**示例**：
- 用户查询北京 3 次后 → 系统主动推荐："您已查询北京 3 次，推荐希尔顿酒店"

### 常见面试问题与答案

#### Q1: 为什么不完全依赖 LLM 工具调用，而是构建复杂度评估框架？

**A**: 通义千问等弱模型在多工具场景下工具调用成功率为 0-30%。通过实现混合复杂度评估（80% 基于规则 + 20% LLM），我们实现了 100% 工具调用率和代码控制的工作流。基于规则的分类在 <1ms 内处理 80% 的情况，使用意图计数和关键词匹配，仅在边缘情况调用 LLM。这提供了生产稳定性，同时保持了复杂查询的智能性。权衡是需要更多前期工程，但对于弱模型的可靠性至关重要。

#### Q2: 详细说明三路混合检索架构和 RRF 融合公式

**A**: 三条检索路径并行运行：(1) BM25 稀疏检索，使用 jieba 中文分词进行精确关键词匹配，(2) 使用 DashScope 1536 维嵌入对原始查询进行稠密语义搜索，(3) 对 LLM 重写查询进行稠密搜索以改进召回率。RRF 融合公式为 score(doc) = Σ weight_i / (k + rank_i)，其中 k=60 是平滑因子，rank_i 是文档在路径 i 中的位置，权重均为 1.0 以实现平等贡献。这为出现在多条路径和顶部位置的文档提供更高分数。查询重写通过标准化口语化术语增加 20-30% 召回率提升。最终准确率：80% vs 单路径 50-60%。

#### Q3: 任务分解如何处理依赖关系并防止死锁？

**A**: TaskDecomposer 使用 LLM 生成带显式 depends_on 字段的 JSON 任务列表，由 Pydantic 验证。执行前，我运行基于 DFS 的环检测来捕获循环依赖。拓扑排序将任务分组到执行批次中，每个批次仅包含依赖关系已满足的任务。批次内，任务使用 asyncio.gather() 并行执行。批次间，执行是顺序的。例如，"查询客户地址然后规划路线"创建 2 个批次：批次 1 有客户查询（无依赖），批次 2 有路线规划（依赖批次 1）。这节省 50% 时间 vs 顺序执行，同时防止死锁。

#### Q4: 解释三层记忆系统及其如何实现个性化

**A**: Layer 1 (ChatMemory)：文件持久化，20 条消息滑动窗口，提供即时对话上下文。Layer 2 (WorkingMemory)：内存实体提取（城市、酒店、日期），30 分钟 TTL，实现会话内实时上下文感知。Layer 3 (LongTermMemory)：JSON 用户画像存储频率模式（如 'Beijing': 3 次访问，'Hilton': 2 次预订），对话结束时增量更新。MemoryService 门面将所有三层融合到增强提示中。示例：3 次北京查询后，系统主动建议"您已查询北京 3 次，推荐希尔顿酒店"。符合 GDPR，支持用户数据删除。

#### Q5: LangSmith 可观测性与传统日志相比如何？给出具体调试示例

**A**: 传统日志需要手动插桩、日志解析，无法可视化调用链。LangSmith 需要 3 个环境变量（LANGCHAIN_TRACING_V2=true、API 密钥、项目名称），无需代码修改即可自动追踪所有 LangChain 调用。具体示例：用户报告"答案不准确"。使用日志，我需要添加调试语句、重新部署、重现问题、解析日志 - 需要数小时。使用 LangSmith，我点击失败追踪，查看完整调用树，检查检索器输出，发现它返回了错误文档 - 5 分钟找到根本原因。还提供 token 使用、成本追踪和每个组件的延迟分解。调试时间减少 95%。

## 🛠️ 技术栈

### 核心框架
- **LangChain 0.1.0+** - 编排框架、链组合、工具抽象

- **Pydantic 2.5.0+** - 数据验证、JSON 解析、类型安全
- **Python 3.10+** - async/await、类型提示、现代语法

### 智能体层
- **ChatTongyi** - 阿里通义千问 LLM（主要推理引擎）

### 可观测性
- **LangSmith** - 自动追踪、性能监控、成本追踪、调试

- **FAISS** - 内存向量存储（RAG）
